# 週報テンプレート設計

社内に既存の週報テンプレートは存在しない。ゼロから「AIが解析することを前提にした週報」として設計する。

## 設計原則

1. **5分で書ける**。設問は7つ、必須は3つだけ。書く負担が高いと提出率が下がり、全機能のデータ源泉が枯れる
2. **構造化はAIの仕事、人は自由に書く**。選択式フィールドは最小限にし、自由記述からAIがスキル・コンディション・相談ごとを抽出する
3. **書くと得をする**。週報を書く → 経歴書が育つ / 詰まりごとにメンターが反応する、という即時リターンを返す
4. **本音が書ける**。コンディション欄の閲覧範囲を明示し、評価に使わないことを宣言する

## テンプレート（フォーム項目）

| # | 項目 | 形式 | 必須 | 主な下流利用 |
|---|---|---|---|---|
| 1 | 今週のコンディション | 選択: ☀️ 好調 / 🌤 普通 / ☁️ モヤモヤ / 🌧 しんどい | ✅ | コンディション検知（自己申告系統） |
| 2 | 稼働の体感 | 選択: 余裕あり / ちょうどいい / 忙しい / 限界が近い | ✅ | コンディション検知（負荷シグナル） |
| 3 | 今週やったこと | 自由記述（箇条書き推奨） | ✅ | スキル抽出、経歴書の実績蓄積 |
| 4 | 新しく触れた技術・初めてやったこと | 自由記述 | – | **スキルマップ更新の最重要ソース** |
| 5 | 詰まったこと・モヤモヤしていること | 自由記述 | – | AIメンター提案、コンディション解析 |
| 6 | 来週やること・やりたいこと | 自由記述 | – | 成長目標のトラッキング |
| 7 | 会社への共有・相談 | 自由記述 + ☑「営業に直接相談したい」 | – | 即時アラート（チェック時は解析を待たず通知） |

### 4段階（偶数）にする理由
5段階だと「3: 普通」に回答が集中して信号が消える。偶数にして中央を排除し、良い/悪いのどちらかに倒してもらう。

### 選択式を2問だけにする理由
自己申告スコアはAIトーン解析と独立した系統として毎週必ず取れる唯一のデータ。逆にスキルや工程を選択式にすると入力が億劫になり形骸化するため、技術情報はすべて自由記述 + AI抽出に寄せる。

## AI解析パイプライン（提出時に非同期実行）

```
提出 → マスキング前処理（顧客名・固有名詞の検出）
     → ① スキル抽出: 設問3,4から技術・工程・役割を構造化 → 既存スキルマップと差分 → SkillSuggestion
     → ② トーン解析: 設問3〜7の文章からコンディションスコア算出 → 自己申告との乖離・トレンドを記録
     → ③ 相談検知: 設問5,7から要対応シグナル → 閾値超えでアラート
     → ④ フィードバック生成: 「今週の成長ポイント」を本人に返す（書くと得をする体験）
```

### 乖離シグナル
自己申告（☀️）とAIトーン解析（低スコア）が乖離しているケースは「弱音を書けない状態」の可能性があり、単独のスコア低下より重要なシグナルとして扱う。

## 運用設計

- **提出サイクル**: 毎週金曜〜月曜朝。未提出者にはリマインド（それ自体もコンディションシグナル: 連続未提出は不調の兆候）
- **下書き自動保存**: 週の途中でメモ的に書き足せる
- **閲覧権限**:
  - 設問1,2,5とコンディションスコア → 本人 + ADMIN + 担当営業のみ
  - 設問3,4（実績系） → 上記 + 経歴書経由で営業活動に利用
- **同意**: 初回オンボーディングで「AIが解析すること」「閲覧範囲」「評価に使わないこと」を明示して同意を取る
